## 💡 Was ich gelernt habe

### Public vs. Private Tabellen-Trennung
```lua
local WeaponModule = {}  -- public: das gibt require() zurück

local weapons = {        -- private: nur intern sichtbar
    {name = "Common", damage = 10, weapons = {"Knife", "Gun", "Sword"}},
    ...
}

function WeaponModule.getRandom(rarity) ... end

return WeaponModule
```
Daten und Interface immer trennen — sonst iteriert Luau beim Loop auch über die Funktionen und wirft einen Fehler.

**C#-Vergleich:** `private` Felder vs. `public` Methoden in einer Klasse.

### Nested Tables (Tables in Tables)
```lua
local weapons = {
    {name = "Common", damage = 10, weapons = {"Knife", "Gun", "Sword"}},
}
-- Zugriff auf inneres Array:
local randomWeapon = entry.weapons[math.random(1, #entry.weapons)]
```

### Mehrere ModuleScripts verknüpfen
`WeaponModule` ruft intern `RarityModule.getRandom()` auf — ein Modul nutzt ein anderes. Das ist der Kern professioneller Spielarchitektur.

### Rückgabe eines Tables statt eines einzelnen Werts
```lua
return {name = randomWeapon, damage = weapon.damage}
-- gibt ein komplettes Objekt zurück, nicht nur einen String
```
**C#-Vergleich:** Wie ein anonymes Objekt `new { Name = x, Damage = y }` oder ein DTO.

## 🐛 Bugs die ich selbst gefixed habe

- Großschreibung: `"common"` vs `"Common"` → Luau unterscheidet Groß/Kleinschreibung beim Vergleich, Namen in allen Modulen angeglichen
- Funktion nach `return` platziert → vor `return WeaponModule` verschoben
- Loop über `AuraModule` statt über private `auras`-Tabelle → Funktion wurde mititeriert → `attempt to index function with 'name'`

## 🔗 Vergleich zu C#

| Luau | C# |
|---|---|
| `local Module = {}` (leer) + `local data = {}` | `public class` + `private field` |
| `return {name = x, damage = y}` | `return new { Name = x, Damage = y }` |
| Zwei ModuleScripts die sich gegenseitig `require()` | Zwei Klassen die sich gegenseitig nutzen |
