## 💡 Was ich gelernt habe

### LocalScript vs. Script (Client vs. Server)
```lua
-- Script → läuft auf dem SERVER
-- ✅ DataStore, Spielwelt für alle verändern
-- ❌ kein GUI

-- LocalScript → läuft beim SPIELER (Client)
-- ✅ GUI steuern, Button-Klicks registrieren
-- ❌ kein DataStore
```
Alles was das Spiel-Ergebnis beeinflusst → Server. Alles was nur angezeigt wird → Client.

**C#-Vergleich:** LocalScript = JavaScript im Browser (Frontend), Script = ASP.NET Controller (Backend)

### GUI per Code erstellen mit Instance.new()
```lua
local function create(className, properties, parent)
    local instance = Instance.new(className)
    for key, value in pairs(properties) do
        instance[key] = value
    end
    instance.Parent = parent
    return instance
end

-- Verwendung:
local frame = create("Frame", {
    Size = UDim2.new(0, 360, 0, 300),
    BackgroundColor3 = Color3.fromRGB(12, 12, 12),
}, rollGui)
```
DRY-Prinzip: eine Helper-Funktion statt 10 Zeilen pro Element.

**C#-Vergleich:** `Instance.new()` = `new Button()` in WinForms/WPF

### ProximityPrompt (Press E Interaktion)
```lua
local proximityPrompt = mainPart:WaitForChild("ProximityPrompt")
proximityPrompt.ActionText = "Roll Aura"
proximityPrompt.KeyboardKeyCode = Enum.KeyCode.E

proximityPrompt.Triggered:Connect(function()
    rollGui.Enabled = true
end)
```
ProximityPrompt muss in einem **BasePart** stecken (nicht direkt im Model) damit Roblox weiß wo er ihn anzeigen soll.

### WaitForChild vs. FindFirstChildOfClass
```lua
-- WaitForChild: sucht direkte Kinder, wartet bis sie existieren
rollMachine:WaitForChild("MainPart")

-- FindFirstChildOfClass(class, true): sucht rekursiv durch alle Nachkommen
rollMachine:FindFirstChildOfClass("ProximityPrompt", true)
```
